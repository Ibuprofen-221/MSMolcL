# 2.5 history_store.py 无文件锁保护 — 解决方案

## 问题定位

`backend/services/history_store.py` 中历史记录的读写流程为典型的 read-modify-write 模式，无任何并发保护：

```python
# backend/services/history_store.py:105-148

def upsert_task_record(task_id, user_data_path, ...):
    records = _read_data(user_data_path)      # 1. 读取整个 JSON 文件
    # ...查找或创建 target 记录...
    _atomic_write(user_data_path, records)     # 3. 写回整个文件

def delete_task_record(task_id, user_data_path):
    records = _read_data(user_data_path)       # 1. 读取
    # ...过滤掉目标记录...
    _atomic_write(user_data_path, records)     # 3. 写回
```

竞争场景：

```
时间 →
请求A: _read_data() ──── 修改 records ──── _atomic_write()
请求B:          _read_data() ──── 修改 records ──── _atomic_write()
                                          ↑
                                  请求 B 的写入覆盖了请求 A 的变更
```

`_atomic_write` 本身是原子的（写 tmp 文件 + rename），但 **read-modify-write 整个流程不是原子的**。

## 影响

- 同一用户的两个并发请求（如：前端同时触发检索 + 历史更新）会导致写丢失
- 历史记录状态可能不准确

## 解决方案

### 方案选择

**方案 A**：使用 `threading.Lock` + 以 `user_data_path` 为键的锁字典（轻量，适合单进程）

**方案 B**：使用 `fcntl.flock` 文件锁（适合多进程）

当前部署为 `uvicorn` 单进程 + 多线程，选择方案 A。

### Step 1: 实现用户级锁管理器

```python
# backend/core/user_lock.py（新建）

from threading import Lock


class UserLockManager:
    """按用户 data_path 维度管理写锁，防止同一用户并发写入历史记录。"""

    def __init__(self):
        self._global_lock = Lock()
        self._locks: dict[str, Lock] = {}

    def acquire(self, user_data_path: str) -> Lock:
        """获取指定用户的锁（不存在则创建）。"""
        # 简短锁定以操作字典
        with self._global_lock:
            lock = self._locks.get(user_data_path)
            if lock is None:
                lock = Lock()
                self._locks[user_data_path] = lock
        return lock


user_lock_manager = UserLockManager()
```

### Step 3（可选）: 改造为真正的小型数据库

当单用户历史记录超过数百条时，JSON 文件本身的读写开销也会成为瓶颈。此时的长期方案是将历史记录迁移到 SQLite（项目已有 SQLAlchemy 基础设施）：

```python
# 新增模型 backend/models/task_history.py
from sqlalchemy import Column, Integer, String, DateTime
from core.db import Base

class TaskHistory(Base):
    __tablename__ = "task_history"

    id = Column(Integer, primary_key=True)
    username = Column(String(64), index=True, nullable=False)
    task_id = Column(String(64), index=True, nullable=False)
    normal_status = Column(String(16), default="pending")
    advanced_status = Column(String(16), default="pending")
    note = Column(String(512), default="")
    created_at = Column(DateTime, nullable=False)
```

数据库层面天然支持事务和行级锁，彻底解决读写竞争问题。此项可作为「后期升级」任务单独执行。
