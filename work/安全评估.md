## 二、已具备的防护能力

### 认证层

- 密码使用 bcrypt 哈希（passlib），不可逆
- JWT HS256 签名，24h 过期
- 前端 token 存 localStorage（仍有 XSS 窃取风险，见问题 1.3）

### 登录防爆破

`backend/api/auth.py` 已实施 LoginGuard（问题 1.2）：

- `LoginGuard` 按 `(IP, username)` 维度计数失败次数
- 5 次失败 / 300s 窗口 → 封禁 600s
- 超过 10000 条记录自动清理最旧条目
- 配合 `@limiter.limit("20/minute")` 形成双层防护

### 速率限制

slowapi（`core/rate_limit.py`）：

- 限流规则按端点分类：auth register (10/min)、auth login (20/min)、其他有保护端点 (60/min)
- IP 识别从 `X-Forwarded-For` / `X-Real-IP` 头获取

### 文件操作安全

- 上传：扩展名白名单（mgf/txt/json）、单文件 10MB、总数 50
- 下载/读取：文件名白名单（`statas.json` 等 4 个固定名）→ 杜绝任意文件下载
- 路径穿越：`_validate_path()` 使用 `resolve()` + `startswith()` 校验在用户目录内
- 原子写入：`tmp → os.replace()` 防止写一半被读到

### 输入校验

- Pydantic 模型校验请求体（`RegisterRequest`、`LoginRequest`、`SmilesBatchPayload` 等）
- SMILES 渲染：单条 ≤500 字符、批量 ≤50、`asyncio.timeout(5s)`
- SQL 注入：SQLAlchemy ORM 参数化查询，无原生 SQL 拼接

## 三、仍然存在的安全缺口

### 🔴 急需修复

| # | 问题 | 受影响端点 | 攻击场景 |
|---|------|-----------|---------|
| 1 | **无认证 + 无限流** | `/health`、`/api/docs-content` | 可被用于 DDoS 探测；`/api/docs-content` 读取本地文件暴露系统文档内容 |
| 2 | **无输入大小限制** | `PUT /api/my-data` | 可上传无限大 JSON 撑爆磁盘 |
| 3 | **多端点缺限流** | `/api/upload-files/status`、`/api/statas`、`/api/history` 系列、`/api/download-file`、`/api/spectrum/plot`、`/api/candidates_choose` 等 | 已认证用户可高频调用消耗服务器资源 |
| 4 | **限流器 IP 伪造风险** | 所有有限流的端点 | slowapi 从 `X-Forwarded-For`/`X-Real-IP` 取 IP，若反向代理未正确剥离这些头，攻击者可伪造 IP 绕过限流 |
| 5 | **JWT 默认密钥** | 所有需要认证的端点 | `jwt_secret_key = "please-change-this-secret-in-production"`，如果生产环境未覆盖环境变量，任何人可伪造 token |

### 🟡 建议加固

| # | 问题 | 说明 |
|---|------|------|
| 6 | **无请求体大小限制** | FastAPI 默认无全局 body 限制，攻击者可发送超大 body |
| 7 | **无并发连接限制** | 无连接数或并发请求数限制（需反向代理层处理） |
| 8 | **CORS 过于宽松** | `cors_allow_methods=["*"]`、`cors_allow_headers=["*"]` — 虽然 origin 白名单有效，但 method/header 的 wildcard 增加了攻击面 |
| 9 | **无审计日志** | 登录成功/失败、文件下载、数据修改等敏感操作无审计记录，出事无法溯源 |
| 10 | **`/api/docs-content` 信息泄露** | 将 `文档说明.md` 结构化后返回，暴露系统内部文档 |

## 四、攻击抵御能力评估

| 攻击类型 | 抵御能力 | 说明 |
|---------|---------|------|
| 暴力破解登录 | ✅ **强** | bcrypt 慢哈希 + LoginGuard IP/用户名计数 + 20/min 限流，三重防护 |
| 批量注册垃圾账号 | ⚠️ **中** | 10/min 限流可缓解但无法阻止；无验证码/邮箱验证 |
| DDoS / CC 攻击 | ❌ **弱** | 仅应用层 slowapi，无 WAF、无连接数限制；`/health` 裸奔 |
| SQL 注入 | ✅ **强** | SQLAlchemy ORM 全参数化，无原生 SQL |
| 路径穿越 / 任意文件读取 | ✅ **强** | 文件名白名单 + `resolve()` + 用户目录前缀校验 |
| CSRF | ⚠️ **中** | JWT 存在 Authorization header 中（非 Cookie），天然防 CSRF；但无显式 CSRF token |
| XSS → Token 窃取 | ❌ **弱** | Token 存 `localStorage`，任何 XSS 可读取（见问题 1.3）|
| JWT 伪造 | ⚠️ **中** | 依赖环境变量覆盖默认密钥，若未设置则可被伪造 |
| IP 伪造绕过限流 | ⚠️ **中** | 取决于反向代理是否正确配置了 header 剥离 |
| 恶意文件上传 | ✅ **强** | 扩展名白名单 + 大小限制；上传内容不会被解释执行 |
| 资源耗尽（磁盘） | ❌ **弱** | 无用户配额（见问题 4.1）；`/api/my-data` PUT 无 body 大小限制 |

---

## 五、优先修复建议

按投入产出比排序：

1. **给 `/health` 和 `/api/docs-content` 添加限流**（5 分钟工作量）
2. **给 `/api/my-data` PUT 添加 body 大小限制和 JSON 深度限制**（10 分钟）
3. **确保 `JWT_SECRET_KEY` 在生产环境被覆盖**（运维层面）
4. **给所有缺限流的认证端点统一加一个适度的限流**（如 120/min，统一定义变量）
5. **检查反向代理是否正确剥离 `X-Forwarded-For` 等头**，或修改 `rate_limit.py` 优先信任 `request.client.host`

第 4 项涉及的端点共 12 个（`/api/upload-files/status`、`/api/statas`、`/api/history` 3个、`/api/candidates_choose`、`/api/candidate_databases`、`/api/download-file`、`/api/spectrum/plot`、`/api/plot`、`/api/my-data` 2个），目前都是"认证了就不限速"的状态。
