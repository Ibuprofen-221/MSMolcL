## 目标
将普通检索从单库 `pubchem` 升级为**可多选数据库并并行检索**，同时把 Top100 结果升级为结构化明细，支持详情页展示完整字段。

## 已落地实现（2026-04）

### 1) 多数据库选择与透传
- 后端新增接口：`GET /api/candidate_databases`
  - 动态扫描 `retrieve_database_root`（默认 `/root/autodl-tmp/database`）下所有 `*.parquet`
  - 返回：`available_databases`、`default_databases`
- 候选池接口 `POST /api/candidates_choose`
  - 新增 `databases` 参数（支持 JSON 数组字符串或 CSV）
  - 服务端做有效性校验与默认值回填
  - 返回数据中包含 `databases` 和 `available_databases`
- 检索提交链路已打通：
  - `candidates_choose -> /api/retrieve -> retrieve_service -> retrieve.py`
  - `databases` 全链路透传，旧请求无该参数时默认回落 `pubchem`

### 2) 两阶段检索（低内存）
普通检索 `retrieve.py` 的 `pubchem` 分支已改为多库两阶段：

- **阶段A：候选池构建（轻量）**
  - 对每个谱图按 `neutral_mass ± ppm` 在所选数据库扫描质量窗口
  - 仅读取 SMILES 列并去重（不缓存全量元信息）
  - 全谱图候选汇总后统一分子编码

- **阶段B：Top100 延迟元数据回查**
  - 对每个谱图最终 Top100 SMILES，回查所选数据库窗口内元数据
  - 聚合字段：
    - `formula`、`inchi_key`：优先取首个非空
    - `generic_name` / `database_name` / `database_id`：跨库去重汇总

该策略避免了在 10ppm 候选阶段构建超大元数据字典，显著降低内存与 IO 压力。

### 3) 结果结构升级与兼容
普通检索与高级检索结果均新增：

- `result_top100`: `list[object]`
  - `rank`
  - `score`
  - `smiles`
  - `formula`
  - `generic_name`
  - `database_name`
  - `database_id`
  - `inchi_key`
  - 以及列表化字段：`generic_name_list` / `database_name_list` / `database_id_list`

同时保留旧字段：`top1`、`top10`、`top100`、`top100_score`，确保历史页面与旧逻辑可继续运行。

### 4) 高级检索兼容改造
- `retrieve_advanced.py` 读取候选池时：
  - 优先从 `result_top100[*].smiles` 取候选
  - 回退到旧 `top100`
- 高级检索输出也写回 `result_top100`
- 失败场景也补齐空 `result_top100`，保证字段稳定

### 5) 前端改造

#### 检索页（`ChemSearch.vue`）
- 新增数据库多选控件（仅 `pubchem` 模式显示）
- 页面初始化时拉取候选数据库列表
- 提交候选池时附带 `databases`
- 结果展示与“生成 top10 图片”逻辑统一改为：
  - 优先读 `result_top100`
  - 回退旧 `top100/top100_score`

#### 高级检索页（`AdvancedSearch.vue`）
- 同步兼容 `result_top100` 与旧字段

#### 详情页（`SpectrumDetailCard.vue`）
- 从“仅 Top10 smiles 文本”升级为“**结果明细卡片**”：
  - 左侧固定尺寸分子图
  - 右侧完整字段（rank/score/smiles/formula/generic_name/database_name/database_id/inchi_key）
  - 列表区域独立滚动
- 支持分别加载普通/高级结果结构图
- 仍保留 Top10 与 Top100 排名查询能力

## 迁移期字段约定（重要）
前端与后端应遵循：
1. **读取优先级**：优先 `result_top100`，缺失再回退 `top100/top100_score`。
2. **写入要求**：新任务必须写 `result_top100`；旧字段保留到全量迁移完成。
3. **高级检索依赖**：高级检索候选读取同样遵循该优先级，确保新旧数据都可跑通。
