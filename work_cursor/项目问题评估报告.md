### 1.2 登录/注册接口无限流保护

**现状**：已**修复且优于文档方案**。
- `core/auth_guard.py` 实现了 `LoginGuard`：按 (IP, username) 维度计数失败，5 次/300s 触发 600s 封禁，并有上限 10000 条+定期清理。
- `auth.py` 中 `register_api` / `login_api` 均使用 `@limiter.limit(rate_limit_auth_login_rule | rate_limit_auth_register_rule)`，对应配置默认 `20/minute`、`10/minute`。
- IP 提取通过 `get_request_client_ip` 优先读取 `X-Forwarded-For`、`X-Real-IP`，回退 `request.client.host`。

**文档方案合理性**：✅ 完全合理。当前实现比文档更细致（窗口清理、字典上限），无需调整。

**遗留风险**：
- `LoginGuard` 是单进程内存态，如果将来部署多进程（例如 `gunicorn -w N`），各进程之间数据不共享。建议保留"单进程 uvicorn"约束或将来迁到 Redis。
- `rate_limit_ip_header_priority` 默认信任代理头，**前置反代必须正确剥离/重写这些头**，否则会被攻击者绕过（这条在 `安全评估.md` §三 也提到了，仍未在代码层加防御）。

### 1.5 `torch.load(weights_only=False)` 任意代码执行

**现状**：⚠️ **仍存在**。`grep` 显示当前共有 7 处 `torch.load` 全部带 `weights_only=False`：
- `services/retrieve.py:658`
- `services/retrieve_advanced.py:106`
- `services/model/predict.py:223`
- `services/model/dataset.py:315, 320`
- `services/model/train.py:434`
- `services/model/test_predict.py:559`
- `services/model/test_predict_advanced.py:97`

**文档方案合理性**：⚠️ 部分合理。文档建议"先尝试 `weights_only=True` 失败再回退" + 哈希清单校验，但有以下问题：
1. `weights_only=True` 在 PyTorch 2.6+ 之前对部分自定义类（如 `ConfigDict`）是直接 `UnpicklingError` 而非可控失败，回退路径几乎一定会被走到，"先尝试"形同虚设；
2. checkpoint 中存储了 `config`（自定义 `ConfigDict`/`dict`），实际不可能完全用 `weights_only=True` 加载；
3. 哈希清单方案需要训练侧配合维护，有运维负担。

**改进方案（建议替代文档方案）**：
1. **方案 A（推荐）**：保持 `weights_only=False`，但严格控制路径来源——`MODEL_WEIGHT_PATHS` / `ADV_MODEL_WEIGHT_PATHS` 当前已经是从 `core/config.py` 硬编码读取（`autodl-tmp/*.pth`），不接受用户输入。在此前提下，**真正的修复点是确保 `autodl-tmp/` 目录权限收紧**（部署侧检查清单），而不是改 Python 代码。
2. **方案 B**：将 `state_dict` 与 `config` 拆分存储——`state_dict` 用 `torch.save(model.state_dict(), p)` + `torch.load(p, weights_only=True)`；`config` 单独写一个 `*.json`。这是治本但成本高的方案。
3. **方案 C（折中）**：在 `services/retrieve.py` 中增加 `safelisting`：`torch.serialization.add_safe_globals([ConfigDict])`，然后 `weights_only=True`。这是 PyTorch 2.4+ 的官方推荐做法。

**结论**：文档方案不必照抄，建议先做方案 A 的"路径不接受外部输入审计"，再视模型存储改造可行性选 C。

### 1.8 TaskID 可猜测导致越权访问

**现状**：🟡 文档描述的威胁**实际上不成立**，但有**真实存在的变体**。

**当前代码分析**：
- `_ensure_task_belongs_to_user(current_user.data_path, task_id)` 调用 `get_user_task_dir(user_data_path, task_id, ...)`，返回路径 `user_data_root / current_user.data_path / task_{task_id}`。
- 用户 A 的 `data_path` 是其用户名（注册时由 `to_user_data_relative_path(username)` 生成），用户 B 的 task_id 在 A 的目录下不存在 → `task_dir.exists()` 返回 False。
- 所以"另一个已认证用户拿着 B 的 task_id 即可访问 B 的数据"——**这条威胁不成立**。

**真实存在的变体（需要修复）**：

> "认证用户访问自己以外的任务通过路径无效路径返回"——这部分 OK；但**部分接口完全跳过了 `_ensure_task_belongs_to_user`**，只检查"目录存在"，且这些接口的 `task_id` 由用户输入：

| 端点 | 是否调 `_ensure_task_belongs_to_user` | 备注 |
|------|------------------------------------|------|
| `/api/retrieve` | ❌ 仅 `_validate_entry_files()`（检查文件存在）| 见 `retrieve.py:354-361` |
| `/api/retrieve-advanced` | ❌ 同上 | 见 `retrieve.py:471-478` |
| `/api/retrieve/status`、`/api/retrieve-advanced/status` | ✅ 有 | OK |
| `/api/upload-files/status` | ❌ 仅根据 `task_dir.exists()` | `sirius_batch_service.py:351` |
| `/api/spectrum/plot` 与 `/api/spectrum/plot`（detail） | ❌ 仅路径校验 | OK 因为路径基于 `current_user.data_path` 拼接 |
| `/api/statas`（带 `path` 参数） | 🟡 `_validate_path` 检查 `startswith(user_root)`，OK |
| `/api/candidates_choose` | ❌ 仅 `task_dir.exists()` | 与 retrieve 同 |
| `/api/download-file` | ❌ 仅父目录存在性 | 与 retrieve 同 |

由于路径全部以 `current_user.data_path` 为前缀，**A 用户即使提交 B 的 task_id 也只会拼到 `user_data_root/A/task_xxx`**，找不到目录。所以"已认证用户跨用户越权"实际**不成立**。

**真正的安全收益场景**有 2 个：
1. 防止"内部权限误用"——比如未来加管理员等多角色，避免漏控；
2. 配合**审计日志**——记录"用户 A 尝试访问 task_id=xxx 但失败"。

**建议**：文档的 `task_owner.json` 方案过重（涉及 7 个接口、所有 task 创建点的改造、向后兼容），收益有限。**优先级可下调**，仅在引入审计/角色后再实施。

### 2.5 history_store.py 无文件锁保护

**现状**：❌ 已修复。`services/history_store.py:121-122, 159-160` 均用 `user_lock_manager.acquire(user_data_path)` 包裹整个 read-modify-write。

**文档方案合理性**：✅ 合理（且已采纳）。

**改进点（独立发现）**：
1. `core/user_lock.py:11-20` 的 `acquire` 实现有**轻微竞态**：第一次 `lock = self._locks.get(key)` 在 `_global_lock` 之外，看到 None 后再进锁拿 lock。最坏情况是同一个 key 在并发下短暂创建两把锁中的一把后被丢弃——实际不会泄露（GC 会回收），但理论上不优雅。第二次进锁时再 `get` 是 OK 的，结论：现实表现没问题，但可读性差，可考虑直接全程持 `_global_lock`（只是个 dict 操作，开销可以忽略）。
2. `delete_task_record` 把 `_remove_task_dir_if_exists`（`shutil.rmtree`）放在锁内，目录大时持锁时间长会阻塞其他历史更新。建议把"目录删除"挪到锁外（先锁内删 records，记录返回值，锁外做 rmtree）。
3. `list_user_records` 是只读路径但**没拿锁**。如果它和 `_atomic_write` 并发，由于 `_atomic_write` 用的是 `tmp.replace()`（POSIX 原子 rename），读到的数据要么是旧版要么是新版，不会损坏，OK。但如果未来改成"非原子写"就要回头加锁。

### 2.6 GPU Worker 进程静默崩溃检测滞后

**现状**：⚠️ **仍存在**。`retrieve_runtime.py:306-314` 仍是只在 `submit()` / `get_job()` 时通过 `_sync_worker_state()` 同步检测；无后台心跳线程。`retrieve_advanced_runtime.py:304-312` 同。

**文档方案合理性**：⚠️ 大方向合理，但**自动重启需谨慎**：
- 文档建议在 `_health_loop` 里 `_restart_worker(ion_mode)` 自动拉起新进程。但 GPU 进程崩溃通常意味着**显存损坏 / CUDA context 断裂 / OOM**，简单重启很可能再次崩溃，形成"崩溃-重启-崩溃"循环。
- 模型重新加载耗时 10-30 秒，重启期间用户感受到"任务全部失败但又显示在跑"。

**改进方案**：
1. **第一步（必做）**：加后台心跳线程，仅做"检测 + 标记 jobs 失败 + 记录 error 日志"，**不自动重启**；
2. **第二步（可选）**：把"重启"暴露为运维 API（带认证 + 限频），由人工或 supervisor 决定何时拉起；
3. **第三步（更进一步）**：用 `multiprocessing.Process` 替代 supervisord/systemd 监控不是好方案，建议把 worker 做成独立 systemd unit，用进程管理器原生重启策略 + 后端只做读取队列。

**结论**：心跳线程必要，自动重启不必要（先观测、再重启）。

### 3.1 同步预处理阻塞事件循环

**现状**：❌ 已修复（mgf+json 同步分支）。`api/file_upload.py:234-241` 用 `loop.run_in_executor(None, preprocess_main, ...)`，并用 `asyncio.Semaphore(4)` + `asyncio.gather` 并发处理多对文件（`file_upload.py:270-277`）。

**文档方案合理性**：✅ 合理（且已采纳）。

**遗留**：
- `services/sirius_batch_service.py:241` 内部仍是**同步串行**：`_run_aggregate_if_needed` 在 for 循环里调 `preprocess_main(...)` 串行处理每个 pair。该函数虽然由 `trigger_aggregate_async` 通过 `loop.run_in_executor` 调度到线程池，**整体不阻塞 event loop**，但单个聚合作业的吞吐没有并发化。如果用户上传 30 个 mgf，需要顺序预处理 30 次。改进方案：在 `_run_aggregate_if_needed` 内部用 `concurrent.futures.ThreadPoolExecutor` 并发处理（同样限并发数 4）。
- 默认线程池 (`asyncio.get_running_loop().run_in_executor(None, ...)`) 大小是 `min(32, os.cpu_count() + 4)`。如果 SMILES 渲染、preprocess、聚合同时打满，可能挤占其他 IO 任务。建议显式建一个 `ThreadPoolExecutor(max_workers=4)` 专门跑 CPU 密集任务。

### 3.2 聚合操作阻塞状态轮询请求

**现状**：❌ 已修复。`get_mgf_only_batch_status` 不再同步聚合（`sirius_batch_service.py:405` 注释提示已改异步）。`file_upload_status.py:33-34` 在 `progress.done == progress.total` 且 `not data.files` 时 `await trigger_aggregate_async(...)`。

**注意**：`trigger_aggregate_async` **是 async fn 但内部立即 `return`**（不等真正聚合完成），所以 `await` 它实际等同 fire-and-forget，不阻塞响应。

**文档方案合理性**：✅ 合理（且已采纳）。

**遗留并发缺陷（独立发现，重要）**：

```72:96:web/backend/services/sirius_batch_service.py
_aggregate_tasks: dict[str, asyncio.Task] = {}


async def trigger_aggregate_async(user_data_path: str, task_id: str) -> None:
    """异步触发聚合（不阻塞调用方）。"""
    task_key = f"{user_data_path}:{task_id}"
    if task_key in _aggregate_tasks:
        return

    async def _do_aggregate():
        try:
            task_dir = get_user_task_dir(user_data_path=user_data_path, task_id=task_id, create=False)
            meta = _read_meta(task_dir)
            loop = asyncio.get_running_loop()
            await loop.run_in_executor(
                None,
                _run_aggregate_if_needed,
                user_data_path, task_id, task_dir, meta,
            )
        finally:
            _aggregate_tasks.pop(task_key, None)

    _aggregate_tasks[task_key] = asyncio.create_task(_do_aggregate())
```

问题点：
1. **去重检查不是原子的**：检查 `if task_key in _aggregate_tasks: return`，到 `_aggregate_tasks[task_key] = ...` 之间没有锁。前端 2s 轮询 + uvicorn 多 worker 中断协程切换，**两个协程可能同时穿过这个判断**，并发起 2 次聚合任务（虽然内部 `_run_aggregate_if_needed` 自带"已完成则 return"幂等性，但仍会浪费 CPU/IO）。
2. `_aggregate_tasks` 内存里的数据**没有 TTL 也没有大小上限**，每次新 task 进来一条记录，正常完成后 pop——但如果 `_run_aggregate_if_needed` 抛了异常导致协程异常退出，`finally` 会清理；但**如果 event loop 正在 shutdown** 这条 finally 不一定跑到。
3. 最关键：`_aggregate_tasks[task_key] = asyncio.create_task(...)` **创建了"游离" Task**，没有 reference 跟踪到 lifespan，进程关闭时不会被清理。

**改进方案**：
- 把 `_aggregate_tasks` 替换为 `dict[str, asyncio.Lock]`，用 `async with lock:` 包整个聚合流程；
- 去重判断"是否已聚合"通过 `meta["aggregate_done"]` 而不是任务字典；
- Task 引用集中放在某个 `set()` 里，并在 `lifespan` 关闭时 `gather(..., return_exceptions=True)`。

### 3.3 无 Job 执行超时

**现状**：⚠️ **仍存在**。`retrieve_runtime.py:82-106` 与 `retrieve_advanced_runtime.py:81-105` 仍是 `result = execute_retrieve_job(job_payload)`，无超时。

**文档方案合理性**：⚠️ **不可直接采用**。文档提议"用子进程包一层 + `subprocess.join(timeout)`"。问题：
- 子进程内部要重新调用 `init_retrieve_process(ion_mode)` → 重新 `torch.load` 模型 → GPU 加载耗时 10-30 秒**每个 job**；
- 模型常驻的设计目的就是避免每个 job 都加载，这个方案直接破坏了它；
- 多个子进程并发抢同一张 GPU 会触发 OOM。

**改进方案（推荐）**：
1. **方案 A — 软超时（首选，不需重构）**：在 worker 主循环里维护一个"任务开始时间戳"，consumer 端线程定期检查"running 状态超过 N 秒"的 job，标记为 `failed` 并发送一条 `kill_job` 信号给 worker 进程（或者直接 SIGTERM 整个 worker 让它重启，损失只有当前那个 job）。这是最小改动方案。
2. **方案 B — `signal.alarm` 在 worker 主线程**：进入 `execute_retrieve_job` 前 `signal.alarm(N)`，捕获 `SIGALRM` 抛 `TimeoutError`。仅在 Linux 主线程有效，但 GPU/CUDA 调用对信号响应不好，可能不会立即生效。
3. **方案 C — 真正彻底的方案**：把 GPU 推理改成 batched，每个 inference step 之间检查超时标志。改造成本高。

**结论**：先做方案 A（软超时 + kill worker），等观测到真正卡死的频率再决定是否走 C。
